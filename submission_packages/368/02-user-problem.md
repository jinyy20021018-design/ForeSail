# 用户问题

## 目标用户

直接经手一笔跨境交易的外贸出口企业，以及为其安排订舱/报关/运输的货运代理（货代）公司——多为中小规模团队。他们最在意一笔在途订单会不会被外部变化（船期、港口、海峡绕航、天气、地缘、贸易政策）打乱，却没有专门盯这些外部变化的工具。

## 使用场景

一笔跨境交易正在履行中（已签约、订舱，多为信用证结算、货走海运）。真正影响这笔货的关键信息——船到哪了、港口有没有拥堵、航线上有没有天气/海峡绕航/地缘或贸易政策变化——几乎都在企业外部、散落在各种新闻与数据源里。今天没有一个能自动跟踪这些外部变化、并把它对到本单的工具：外贸与货代团队只能靠人手动去各处收集，再用 Excel 表格登记、靠人眼盯着，效率很低、更新滞后，往往等发现时风险已经落地。

## 当前解决方式

市面上没有面向『对外风险跟踪』的同类产品——现有外贸数字化工具都在解决企业内部效率。团队竞品调研中的两款代表（九科信息、未斯科技）都是把托书/订舱/改单/单证等内部流程自动化、提升出单效率，属于对内的执行自动化，没有一款把外部世界的变化对到一笔在途订单、告诉你该做什么。所以外贸/货代团队现在只能靠人工盯新闻、盯船、盯港口，用 Excel 和邮件拼接判断。

## 具体痛点

- 影响一笔在途订单的外部信息（船期、港口、天气、海峡绕航、地缘/贸易政策）几乎都在企业外部，且没有自动跟踪工具，只能靠人手动去各处收集、用 Excel 和人眼盯，时间紧、容易遗漏。
- 缺少跨域编排层：现有工具各管一段，没人把外部事件转成『这笔单的具体敞口 → 合同义务 → 截止时间 → 动作』并持续推进。
- 外部事件可能冲击 ETA、最迟装运日、LC 交单/有效期、资金占用或价格，人工难以及时把事件与合同条款、期限对齐。
- Incoterm 责任划分与保险覆盖缺口（如战争/海盗/绕航是否在保）需逐单判断，判错会把风险或损失落到错误一方。

## 问题证据

问题定义来自团队前期产品设计；并做了一手用户访谈——与一家跨境电商进口团队的从业者深度交流，对方以真实经历印证核心痛点：『之前有中东的一批货，供应商就死活运不过来，我们就断了好久』，并明确表示『想研究这种风险是不是可以被覆盖和预测，这种情况其实还挺普遍』；对方现状是没有内部风险判断系统、主要靠上游供应商信息临时补救（有风险时才从日韩等渠道扫货）。访谈还带来一个关键印证：进口/跨境电商中介与出口商其实是同一笔交易的买卖两侧，受访者自己提出『你们和商家的订单其实可以视作一笔正常的外贸交易』——正好印证 ForeSail 以单笔 trade Case 为单位、支持买卖双视角的产品模型，并揭示进口/中介是一个同样强需求的用户群。线上 Demo 使用基于 2026 年霍尔木兹危机公开报道整理的带来源合成演示事件；真实危机背景与来源见 backend/app/data/curated_events.json，演示船舶、交易及单笔受阻情节为构造。当前样本仍小（1 场深度访谈），痛点普遍性与付费意愿待扩大样本验证。

## 尚未验证的假设

- 目标客户中以 LC 结算、受最迟装运日/交单期约束的订单占比足够高，值得单独产品守护。
- 外贸团队愿意为『外部风险决策与归属』这一层单独付费，而非并入现有单证执行工具。
