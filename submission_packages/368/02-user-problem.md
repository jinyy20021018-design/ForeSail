# 用户问题

## 目标用户

国际贸易商与进出口企业中处理同一笔交易的多类角色：贸易运营/物流（订舱、船期、港口异常）、贸易融资（信用证、交单、银行沟通）、风险/保险（覆盖与转移）、财资（FX、融资、现金流）、采购/销售，以及承担风险与成本取舍的管理层审批人。

## 使用场景

一笔跨境交易正在履行中（已签约、订舱，多为信用证结算、货走海运）。当航线上出现天气、海峡/运河绕航、港口拥堵、海盗、地缘政治或贸易政策等外部事件时，相关信息分散在合同、PDF、邮件、银行与保险回复及多个外部数据源里，团队需要在短时间内同时判断：这事和本交易相不相关？会落到运输/合同/保险/LC/付款/价格中的哪一环？在什么时间前该向谁发起什么动作？

## 当前解决方式

现有系统各自解决一段：航运平台看船舶与港口状态、CTRM/TMS 管订单与资金、银行平台执行 FX、保险经纪处理保单、贸易融资团队查信用证。真正缺失的是把『外部事件』映射到『这一笔交易该采取什么动作』的跨域编排层——目前主要靠人工在多个系统与邮件之间拼接判断。

## 具体痛点

- 中断发生时需同时处理多条线索（船期、港口、天气、海盗/战争、Incoterm 责任、保险、LC 期限、付款、FX、融资），信息分散在合同/PDF/邮件/银行与保险回复/多个外部数据源，时间紧、容易遗漏。
- 缺少跨域编排层：现有工具各管一段，没人把外部事件转成『这笔单的具体敞口 → 合同义务 → 截止时间 → 动作』并持续推进。
- 外部事件可能冲击 ETA、最迟装运日、LC 交单/有效期、资金占用或价格，人工难以及时把事件与合同条款、期限对齐。
- Incoterm 责任划分与保险覆盖缺口（如战争/海盗/绕航是否在保）需逐单判断，判错会把风险或损失落到错误一方。

## 问题证据

问题定义来自团队《跨境交易中断与风险处置 Agent》项目设计文档 v1.0（2026-06）；并做了一手用户访谈——与一家跨境电商进口团队的从业者深度交流，对方以真实经历印证核心痛点：『之前有中东的一批货，供应商就死活运不过来，我们就断了好久』，并明确表示『想研究这种风险是不是可以被覆盖和预测，这种情况其实还挺普遍』；对方现状是没有内部风险判断系统、主要靠上游供应商信息临时补救（有风险时才从日韩等渠道扫货）。访谈还带来一个关键印证：进口/跨境电商中介与出口商其实是同一笔交易的买卖两侧，受访者自己提出『你们和商家的订单其实可以视作一笔正常的外贸交易』——正好印证 ForeSail 以单笔 trade Case 为单位、支持买卖双视角的产品模型，并揭示进口/中介是一个同样强需求的用户群。主 demo 与线上 demo 挂载 2026 霍尔木兹危机的真实公开报道事件（带来源，见 backend/app/data/curated_events.json）。当前样本仍小（1 场深度访谈），痛点普遍性与付费意愿待扩大样本验证。

## 尚未验证的假设

- 目标客户中以 LC 结算、受最迟装运日/交单期约束的订单占比足够高，值得单独产品守护。
- 外贸团队愿意为『外部风险决策与归属』这一层单独付费，而非并入现有单证执行工具。
